# 求职申请管理看板项目计划

## 一、摘要
本项目定位为一款面向大学生求职季的轻量化申请管理工具，核心目标不是“记录信息”，而是帮助用户在多公司、多岗位、多节点并行推进时，快速回答三个问题：哪些申请最紧急、当前卡在哪一步、接下来该做什么。

文档采用正式 PRD 风格，输出内容以“产品方案 + 路线图”为主，优先覆盖用户痛点、产品结构、核心功能、页面设计、迭代节奏和验收标准，使其既适合面试展示，也能直接作为后续原型设计和开发拆解依据。

## 二、产品目标与设计原则
### 1. 目标用户
目标用户为处于秋招、春招、实习投递阶段的大学生与应届生，典型特征如下：
- 同时投递多个岗位，信息分散在邮件、招聘平台、聊天记录和备忘录中。
- 容易遗漏截止时间、笔试安排、材料补交、面试轮次等关键节点。
- 难以整体判断当前求职进展，心理压力大，复盘效率低。

### 2. 核心问题
产品需要解决以下三类问题：
- 时间管理混乱：不同岗位有不同截止日期和行动节点，靠手动记忆容易遗漏。
- 状态追踪困难：用户不知道每个岗位当前处于哪一轮，是否还需要补材料或准备面试。
- 决策效率低：无法快速看出哪些申请值得优先投入，哪些已经停滞或应放弃。

### 3. 产品目标
- 帮助用户统一管理所有求职申请，形成一处查看的“总看板”。
- 帮助用户以最小认知成本识别优先级和风险。
- 帮助用户从“被动记录”转向“主动推进”，明确下一步行动。

### 4. 设计原则
- 一眼看清全局：首页必须能让用户快速看到申请总数、阶段分布、紧急事项。
- 任务驱动而非表格堆砌：展示方式围绕“接下来做什么”设计。
- 低录入成本：字段尽量精简，优先保证核心信息完整。
- 支持复盘：不仅管过程，还能沉淀申请结果和经验。

## 三、产品定位与价值主张
产品定位为“求职季个人申请指挥台”，区别于普通待办工具或表格管理，重点体现在以下三点：
- 它是面向求职场景设计的，内置申请阶段、材料状态、面试进展等语义。
- 它同时整合看板、日历、提醒、材料和进度追踪，而不是单点记录。
- 它更强调风险提示和推进建议，让用户知道“现在最该处理什么”。

核心价值主张：
- 不漏事：截止时间、面试安排、补材料事项都有提醒与预警。
- 不迷失：每个岗位从投递到结果都有清晰轨迹。
- 不内耗：通过结构化视图降低混乱感，帮助用户维持节奏。

## 四、核心功能方案
### 1. 首页总览 Dashboard
首页是产品的主入口，承担全局感知和行动分发职能，建议包含以下模块：
- 今日待办：展示今天必须处理的事项，如截止投递、笔试、面试、补材料。
- 紧急申请：展示未来 3 天内有关键节点的岗位。
- 进度概览：统计待投递、已投递、笔试中、面试中、Offer、已淘汰等数量。
- 本周时间轴：按日期展示本周的重要安排。
- 风险提醒：标记“即将截止未投递”“长时间无进展”“材料未齐全”等问题。

首页要达到的效果是：用户打开后 30 秒内就能知道自己是否有风险项、近期重心在哪。

### 2. 申请管理看板
这是产品的核心页面，采用看板式分栏呈现，建议默认阶段如下：
- 待投递
- 已投递
- 测评/笔试
- 面试中
- Offer
- 已结束

每张申请卡片需包含：
- 公司名称
- 岗位名称
- 城市/部门（可选）
- 当前阶段
- 申请截止日期
- 下一关键动作
- 材料完成度
- 优先级标签
- 最新更新时间

核心操作包括：
- 新建申请
- 拖拽切换阶段
- 编辑关键节点
- 标记完成某一步
- 快速筛选与搜索

推荐支持的筛选维度：
- 公司 / 岗位关键词
- 当前阶段
- 截止时间
- 优先级
- 是否有风险
- 是否已结束

### 3. 日历与时间管理
考虑到求职场景中时间冲突频繁，需提供独立的时间视图：
- 月视图：看截止日期和重要节点分布。
- 周视图：看面试、笔试、宣讲、补交材料等具体安排。
- 列表视图：按时间先后查看未来事项。

事件类型建议统一为：
- 投递截止
- 网申提交
- 笔试/测评
- 面试
- 补材料
- 跟进提醒
- 结果通知

关键能力：
- 从申请卡片自动生成事件
- 支持手动新增私人提醒
- 支持事件完成/延期
- 支持冲突提示，例如同一天多个面试

### 4. 申请详情页
申请详情页承担“单岗位全生命周期记录”的作用，建议包含：
- 基本信息：公司、岗位、岗位链接、工作地点、薪资范围、来源渠道。
- 进度记录：每次状态变更、面试轮次、反馈结果、时间戳。
- 材料管理：简历版本、作品集、成绩单、证书、求职信等是否已准备/提交。
- 备注区：记录面试官问题、岗位亮点、个人感受、后续准备重点。
- 联系信息：HR 名称、邮箱、微信、电话等。
- 行动清单：当前岗位下一步需要执行的 1~3 件事。

### 5. 材料中心
用户的痛点之一是“不同岗位投不同版本材料”，因此建议设置材料中心：
- 管理常用简历版本和作品集版本
- 记录每个岗位使用了哪一版材料
- 标记某些岗位是否还缺材料
- 查看哪些材料使用频率最高

材料状态建议分为：
- 未准备
- 已准备未提交
- 已提交
- 需更新

### 6. 提醒与风险机制
提醒不是简单通知，而是产品的关键体验点。建议分三类：
- 时间型提醒：截止前 3 天、1 天、当天提醒。
- 进度型提醒：已投递超过 7 天无反馈、面试后超过 5 天无结果。
- 完整性提醒：申请已创建但缺少岗位链接、截止时间、材料状态等关键信息。

风险展示优先级：
- 高风险：24 小时内截止且未完成关键动作。
- 中风险：3 天内有安排，或信息缺失影响推进。
- 低风险：长期无进展，可选择继续跟进或归档。

## 五、关键信息结构与核心对象
为保证后续原型和开发拆解一致，统一以下核心对象定义：

### 1. 申请对象
每条申请至少包含以下字段：
- `company_name`
- `job_title`
- `job_stage`
- `deadline`
- `next_action`
- `priority`
- `application_status`
- `material_status`
- `event_list`
- `notes`
- `updated_at`

### 2. 事件对象
每个事件需包含：
- `event_type`
- `event_date`
- `related_application`
- `status`
- `reminder_time`
- `notes`

### 3. 材料对象
每份材料需包含：
- `material_name`
- `material_type`
- `version`
- `status`
- `used_for_applications`

默认约束：
- 一个申请可以绑定多个事件。
- 一个申请可以绑定多个材料状态。
- 看板阶段与详情页进度记录保持一致。
- 首页统计由申请对象实时聚合生成。

## 六、核心用户流程
### 1. 首次建档流程
用户进入产品后，先录入正在进行的申请：
- 新建申请
- 填写基础信息和截止日期
- 选择当前阶段
- 标记材料情况
- 自动生成提醒和待办

目标是让用户在 5 分钟内完成首批申请初始化。

### 2. 日常使用流程
典型日常流程如下：
- 打开首页查看今日待办和紧急申请
- 进入看板处理当前重点岗位
- 更新某岗位阶段或补充备注
- 在日历中确认未来安排
- 完成事项后关闭提醒

### 3. 单岗位推进流程
当用户收到新的求职进展时：
- 进入申请详情页
- 更新阶段和关键节点
- 记录面试信息或反馈
- 自动刷新首页统计和提醒状态

### 4. 复盘流程
在求职周期后期，用户可以：
- 查看各阶段转化情况
- 总结哪些岗位卡住最多
- 复盘不同材料版本的效果
- 归档结束申请，为下一轮投递提供经验

## 七、页面结构建议
建议产品包含 5 个一级页面：
- 首页总览
- 申请看板
- 日历视图
- 材料中心
- 数据复盘

页面职责划分如下：
- 首页解决“现在最重要的是什么”
- 看板解决“所有申请当前在哪一步”
- 日历解决“什么时候要做什么”
- 材料中心解决“要交什么、缺什么、用了什么版本”
- 数据复盘解决“这一轮投递效果如何”

## 八、迭代路线图
### Phase 1：MVP 版本
目标是完成最小可用闭环，覆盖“记录 + 追踪 + 提醒”：
- 首页总览基础模块
- 申请看板及申请卡片管理
- 申请详情页
- 截止时间与关键节点提醒
- 基础日历视图
- 搜索与筛选

MVP 验收标准：
- 用户可完整录入并管理至少 20 条申请
- 用户可识别所有临近截止事项
- 用户可查看每条申请当前阶段和下一动作
- 用户可在 1 分钟内定位任何一条申请的状态

### Phase 2：增强版本
目标是提升效率和陪伴感：
- 材料中心
- 智能风险提示
- 面试进度记录优化
- 数据统计与投递复盘
- 自定义标签与优先级规则

### Phase 3：差异化版本
目标是形成更强竞争力：
- 投递策略建议
- 面试准备清单模板
- 公司/岗位收藏池
- 多端同步与消息中心
- 与邮箱/日历平台的轻量集成

## 九、验收场景与测试用例
### 1. 核心功能验收
- 新建申请后，可在看板、首页和日历中看到对应信息。
- 拖动申请卡片到新阶段后，阶段统计与详情页同步更新。
- 设置截止日期后，系统按规则生成提醒并在首页显示风险。
- 某申请补齐材料后，材料状态和风险提示同步消失。
- 进入申请详情页后，可完整看到该岗位的历史记录和下一步动作。

### 2. 用户场景验收
- 用户同时管理 10~30 个岗位时，仍能快速识别高优先级事项。
- 用户一周内安排多个笔试/面试时，能在日历中清楚发现冲突。
- 用户长时间未收到反馈时，系统能给出“待跟进”提示。
- 用户结束一轮求职后，能够通过复盘页看到阶段分布和结果沉淀。

### 3. 易用性验收
- 首次录入流程不超过 5 分钟可完成首批初始化。
- 首页主要信息在首屏可读，不依赖深层跳转。
- 申请卡片信息密度适中，不因字段过多影响扫描效率。
- 筛选规则清晰，用户能快速找到“快截止”“面试中”“缺材料”岗位。

## 十、成功指标
建议使用以下指标衡量方案有效性：
- 用户每周主动打开频次
- 新增申请录入完成率
- 临期事项遗漏率
- 高风险岗位处理及时率
- 用户对“求职进展是否更清晰”的主观评分

## 十一、默认假设
- 产品面向个人使用，不考虑团队协作场景。
- 首版以 Web 端为主，保证桌面端体验，移动端以基础适配为准。
- 数据录入默认以手动输入为主，不依赖外部招聘平台打通。
- 提醒方式以站内提醒为默认方案，不强依赖短信或邮件。
- 视觉表达优先采用“看板 + 时间轴 + 风险提示”组合，而不是复杂报表。
- 本次文档目标是完成面试展示级产品规划，不展开数据库、技术架构和工程实现细节。
